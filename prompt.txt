Tu es un ingénieur logiciel senior spécialisé dans les CLI agentiques, la sécurité locale, l’UX terminal, les outils développeur Python et les architectures d’agents IA.

Tu vas travailler sur le projet **Nyx**, un CLI agentique Python situé dans le dépôt courant. Ton objectif est de corriger les problèmes identifiés lors d’un audit, sans refonte inutile, en gardant le projet simple, fiable et maintenable.

## Objectif principal

Rendre Nyx plus crédible comme outil développeur utilisable par d’autres personnes, en priorité sur :

1. la sécurité réelle ;
2. la cohérence des modes agentiques ;
3. la fiabilité des tests ;
4. la propreté du working tree ;
5. la documentation honnête ;
6. l’UX d’onboarding.

Ne cherche pas à ajouter de nouvelles fonctionnalités spectaculaires. Corrige d’abord les garanties existantes.

## Problèmes à corriger en priorité

### 1. Le mode `architect` n’est pas réellement read-only

Problème :
- Le mode `architect` filtre les outils pour ne garder que des outils supposés read-only.
- Mais `repo_map` appelle `build_repo_map()`, qui appelle `index_dependencies()`.
- `index_dependencies()` écrit `.nyx/repo_graph.json`.
- Donc un outil autorisé en mode read-only écrit dans le dépôt.

Fichiers concernés :
- `nyx/agent.py`
- `nyx/repo_map.py`
- `nyx/tools.py`
- tests associés

Correction attendue :
- Rendre `repo_map` sans effet de bord par défaut.
- Ajouter un paramètre explicite du type `write_index: bool = False` si l’index disque est vraiment utile.
- En mode `architect`, garantir qu’aucun outil autorisé ne crée, modifie ou supprime de fichier.
- Ajouter un test qui vérifie qu’un appel à `repo_map` en mode architect ne modifie pas le filesystem.

Critère d’acceptation :
- Un test échoue avant correction et passe après.
- `.nyx/repo_graph.json` n’est pas créé par défaut.

---

### 2. Le changement de mode détruit la liste d’outils active

Problème :
- Dans `Agent._apply_mode_config()`, le mode `architect` filtre `self._all_tools` en place.
- Quand on revient ensuite à `chat`, `code` ou `debug`, les outils retirés ne sont pas restaurés.

Fichier :
- `nyx/agent.py`

Correction attendue :
- Séparer clairement :
  - les outils disponibles de base ;
  - les outils actifs selon le mode.
- Ne jamais muter destructivement la source complète des outils.
- Recalculer la liste active à chaque changement de mode.
- Vérifier aussi les tools des subagents après changement de mode.

Critère d’acceptation :
- Test : démarrer avec tous les outils, passer en `architect`, vérifier la restriction, repasser en `chat`, vérifier que tous les outils reviennent.

---

### 3. Les niveaux `auto` et `yolo` ne restaurent pas proprement les callbacks

Problème :
- `auto` et `yolo` remplacent les callbacks d’approbation.
- Revenir à `ask` ne restaure pas clairement les callbacks précédents.
- Cela peut laisser l’agent dans un état plus permissif que prévu.

Fichier :
- `nyx/agent.py`
- éventuellement `nyx/permissions.py`

Correction attendue :
- Implémenter une gestion explicite de l’état d’autonomie.
- Conserver les callbacks utilisateur originaux.
- Recalculer les callbacks effectifs quand l’autonomie change.
- Revenir à `ask` doit restaurer les prompts.
- Ajouter tests unitaires.

Critère d’acceptation :
- Test : `ask -> yolo -> ask`, une commande dangereuse doit de nouveau demander approbation ou être refusée si aucun callback n’est disponible.

---

### 4. Les tests polluent `.nyx/patches` et `.nyx/rollback`

Problème :
- L’exécution des tests génère ou supprime des fichiers dans `.nyx`.
- Certains artefacts `.nyx/patches` et `.nyx/rollback` sont suivis par git.
- Le test suite ne devrait jamais salir le dépôt.

Fichiers concernés :
- tests autour de `PatchTool`
- `nyx/diff_tool.py`
- `.gitignore`

Correction attendue :
- Modifier les tests pour utiliser systématiquement `tmp_path` ou `TemporaryDirectory`.
- Ne jamais utiliser le dossier projet réel pour rollback/history dans les tests.
- Ignorer les artefacts runtime Nyx qui ne doivent pas être versionnés.
- Évaluer s’il faut retirer du tracking les fichiers runtime déjà suivis, sans supprimer des fichiers utilisateur arbitrairement.

Critère d’acceptation :
- Après `pytest`, `git status --short` ne montre aucun nouvel artefact de test.
- Les tests de rollback/history passent toujours.

---

### 5. Durcir l’exécution shell

Problème :
- `execute_command` utilise `subprocess.run(..., shell=True)`.
- Il existe des gardes, mais la surface d’injection reste élevée.
- Les permissions par défaut sont permissives.

Fichier :
- `nyx/tools.py`
- `nyx/permissions.py`
- `nyx/config.py`
- tests sécurité

Correction attendue :
- Garder la compatibilité utilisateur, mais réduire le risque.
- Si possible :
  - exécuter les commandes simples sans shell via `shlex.split` sur Unix et une stratégie sûre sur Windows ;
  - n’utiliser `shell=True` que pour commandes complexes après approbation explicite ;
  - considérer les opérateurs shell comme nécessitant un prompt ;
  - documenter clairement le comportement.
- Réévaluer les defaults :
  - au minimum, écrire une doc claire ;
  - idéalement, shell default plus prudent que `allow`.

Critère d’acceptation :
- Tests couvrant :
  - commande simple sûre ;
  - commande avec `&&`, `|`, redirection ;
  - commande dangereuse ;
  - comportement sans callback d’approbation ;
  - Windows si possible.

### 6. Encadrer les skills Python

Problème :
- Les skills sont du code Python local exécuté à l’import via `exec_module`.
- C’est acceptable pour un outil hackable, mais dangereux si présenté trop largement.

Fichier :
- `nyx/skill_manager.py`
- README
- config

Correction attendue :
- Documenter explicitement que les skills sont du code trusted.
- Éviter l’auto-chargement silencieux de skills non trusted si possible.
- Ajouter au minimum :
  - logs/avis clair au chargement ;
  - option config pour désactiver les skills ;
  - tests de comportement.
- Si tu peux le faire proprement sans grosse refonte, envisager exécution isolée en subprocess, mais ne bloque pas tout le chantier dessus.

Critère d’acceptation :
- L’utilisateur comprend clairement le risque.
- Les skills peuvent être désactivées.
- Les tests couvrent la désactivation.

### 7. Clarifier ou corriger la promesse de mémoire intelligente

Problème :
- `AgentContext.compress_history()` est désactivé.
- Le README promet une mémoire avec summarisation intelligente.
- `MemoryManager` a une summarisation, mais le comportement réel doit être clarifié.

Fichiers :
- `nyx/agent.py`
- `nyx/memory.py`
- README

Correction attendue :
- Soit réactiver une compression sûre et testée ;
- soit reformuler la documentation pour dire exactement ce qui existe.
- Éviter les promesses vagues.

Critère d’acceptation :
- Tests ou documentation alignés avec le comportement réel.

---

### 8. Ajouter une commande d’onboarding `doctor`

Problème :
- L’utilisateur ne peut pas facilement diagnostiquer son installation.
- `nyx --json -p "hello"` échoue si aucune clé API n’est présente, ce qui est normal mais peu guidant.
- Les outils dev déclarés (`ruff`, `mypy`) peuvent être absents.

Correction attendue :
- Ajouter une commande CLI simple :
  - `nyx doctor`
- Elle doit vérifier :
  - version Python ;
  - présence d’une clé API ou config provider ;
  - provider/model configurés ;
  - présence optionnelle de Rich ;
  - disponibilité de git ;
  - présence de `pytest`, `ruff`, `mypy` si mode dev ;
  - état du dossier projet.
- Elle ne doit pas appeler de LLM.
- Elle doit produire une sortie claire et utile.

Critère d’acceptation :
- Tests CLI pour `nyx doctor`.
- Fonctionne sans clé API.
